# Blood Hunt Companion — Data Pipeline

> **Purpose:** Concrete, runnable instructions for the two data sources the app depends on:
> 1. **Static game data** — extracted offline from the Marvel Rivals install via FModel (UE 5.3 .pak files).
> 2. **Personal gear inventory** — parsed from the user's own screenshots via Tesseract OCR + OpenCV preprocessing.
>
> Companion documents: [`PROJECT.md`](./PROJECT.md) (architecture & features), [`RESEARCH.md`](./RESEARCH.md) (game knowledge & sources).

---

## 0. Why Two Pipelines

Live in-game data is off-limits — NetEase explicitly bans third-party plugins that hook the running process ([Marvel Rivals plugin ban precedent](https://marvelrivals.gg/marvel-rivals-bans-third-party-plugins/)). What *is* available:

| Source | What we get | Method | Live? |
|---|---|---|---|
| Game `.pak` files | Static: hero stats, ability scaling, trait nodes, gear stat ranges, Arcana effects | **FModel + Repak Rivals** | No — re-run after each patch |
| Inventory UI screenshots | The player's own gear pieces with rolled values | **Tesseract OCR + OpenCV** | Player triggers a hotkey/paste |

These two sources are sufficient for every feature in `PROJECT.md`.

## Part 1 — FModel Extraction (Static Game Data)

### 1.1 Tools

| Tool | Purpose | Source |
|---|---|---|
| **FModel** | UE asset viewer/exporter; reads `.pak`, navigates `DT_*` DataTables, exports JSON | [fmodel.app](https://fmodel.app) |
| **Repak Rivals** | Marvel Rivals–specific repacking/mounting helper, kept current with AES keys | [Nexus Mods #1717](https://www.nexusmods.com/marvelrivals/mods/1717) |
| **UE 5.3 mappings** (`.usmap`) | Type info needed for unversioned UE assets | Generated by community; bundled with Repak Rivals or downloadable from FModel community Discord |

### 1.2 Game File Location (Steam install, Windows)

```
C:\Program Files (x86)\Steam\steamapps\common\MarvelRivals\MarvelGame\Marvel\Content\Paks
```

Epic / standalone launcher installs sit under their respective install root; the `…\MarvelGame\Marvel\Content\Paks` suffix is identical.

The folder contains the chunked `.pak` / `.utoc` / `.ucas` files plus a global `*.pak` index. Do **not** edit them. Read-only operations only.

### 1.3 One-Time FModel Setup

1. Install FModel from [fmodel.app](https://fmodel.app).
2. **Settings → Game's Detection** → add a new game:
   - Detection: `Manual`
   - UE Version: **`GAME_UE5_3`**
   - Directory: the Paks folder above.
3. **Settings → Mappings**:
   - Provide the latest `.usmap` file (ships with Repak Rivals on Nexus Mods #1717, or generate one with UE4SS / FModel community tools).
4. **Settings → AES**:
   - Marvel Rivals paks are AES-encrypted. The current AES key is published with each Repak Rivals release on [Nexus Mods #1717](https://www.nexusmods.com/marvelrivals/mods/1717). Paste it under "Main Key" (do not commit this key to the repo).
5. **Load** the game inside FModel. Verify the asset tree expands without errors.

> **Important:** AES keys and mapping files **change on patch days**. Make refreshing them step #1 of any post-patch session.

### 1.4 Asset Targets

Marvel Rivals stores game data in DataTables (`DT_*.uasset`). The targets we care about (exact paths confirmed at first extraction; expect minor name drift between patches):

| Concept | DataTable (likely path under `Marvel/Content/Marvel/...`) |
|---|---|
| Heroes & abilities | `…/Heroes/DT_HeroAbilities.uasset` |
| Trait trees | `…/Heroes/DT_HeroTraits.uasset` |
| Gear stat catalog & ranges | `…/BloodHunt/DT_GearStats.uasset` |
| Gear roll tables | `…/BloodHunt/DT_GearRollTables.uasset` |
| Arcana scrolls | `…/BloodHunt/DT_ArcanaScrolls.uasset` |
| Forge cost & odds | `…/BloodHunt/DT_ForgeRules.uasset` |
| Difficulty scaling (NM 1–160) | `…/BloodHunt/DT_NightmareScaling.uasset` |

If a path differs, search the asset tree for `DT_` + a keyword. FModel's search is fast.

### 1.5 Export Procedure (Per DataTable)

1. Right-click the `.uasset` → **Save Properties (.json)**.
2. Output goes to FModel's configured Output → `Exports/…`.
3. Move the JSON into the project at `data/game/_raw/<name>.json`.

Repeat for each target above. Aim for **complete dumps**, not selective fields — the parser drops what it doesn't need.

### 1.6 Translate Raw Exports → App Schemas

The raw FModel JSON is verbose (UE struct nesting). A small translator script in `tools/translate_game_data.py` flattens each raw export into the canonical shapes in [`PROJECT.md` §10](./PROJECT.md#10-game-data-json-schemas-target-shapes).

```
data/game/
├── _raw/                       ← FModel exports, untouched
│   ├── DT_HeroAbilities.json
│   ├── DT_HeroTraits.json
│   ├── DT_GearStats.json
│   ├── DT_ArcanaScrolls.json
│   └── DT_ForgeRules.json
├── heroes.json                 ← translated, app consumes this
├── traits.json
├── gear_stats.json
├── arcana.json
├── forge_rules.json
└── version.json                ← {"game_version": "20260427.1", "extracted_at": "..."}
```

Translator skeleton:

```python
# tools/translate_game_data.py
import json, pathlib, datetime

RAW = pathlib.Path("data/game/_raw")
OUT = pathlib.Path("data/game")

def load(name): return json.loads((RAW / f"{name}.json").read_text())

def translate_heroes(raw):
    out = []
    # Each top-level row in DT_HeroAbilities is keyed by hero+ability
    for row_name, row in raw[0]["Rows"].items():
        # ... map UE field names to schema fields
        out.append({
            "id": row["HeroId"],
            "ability_id": row["AbilityId"],
            "base_damage": row["BaseDamage"],
            "scaling": [
                {"stat": s["StatId"], "coefficient": s["Coefficient"]}
                for s in row.get("Scaling", [])
            ],
            "cooldown": row["Cooldown"],
        })
    return out

# ... translate_traits, translate_gear_stats, translate_arcana, translate_forge

if __name__ == "__main__":
    (OUT / "heroes.json").write_text(json.dumps(translate_heroes(load("DT_HeroAbilities")), indent=2))
    # ... rest
    (OUT / "version.json").write_text(json.dumps({
        "extracted_at": datetime.datetime.utcnow().isoformat() + "Z",
        # Game version: pull from FModel's package metadata or the user types it
    }, indent=2))
```

## Part 2 — OCR Pipeline (Personal Gear Inventory)

> **Architectural note (2026-04-27):** the pipeline is **calibration-free** and
> content-based. There is no `data/calibration/` folder, no per-resolution JSON,
> no `tools/ocr_calibration.py`. The user takes a full-screen screenshot at any
> resolution; the pipeline does the rest. See PROJECT.md §9 Phase 2 for the
> rationale; this part documents how it works.

### 2.1 Workflow Overview

```
Player takes a full-screen screenshot with a gear tooltip on screen
   │
   ▼
[Stage 1] detect.detect_tooltip       ← Canny+contour, HSV fallback
   │                                     → DetectedCard(bbox, confidence)
   ▼
[Stage 2] anchors.compute_anchors     ← proportional regions inside the card
   │                                     → CardAnchors(name, rarity, level,
   │                                                   slot_icon, base_effect,
   │                                                   extended_effects)
   ▼
[Stage 3] anchors.segment_rows        ← whitespace-projection row split
   │                                     → list[(y_start, y_end)]
   ▼
[Stage 4] _extract_row (per row)      ← preprocess → Tesseract → fuzzy stat
   │                                     → template-match tier
   ▼
[Stage 5] _extract_top                ← name, rarity (color), level, slot,
   │                                     base_effect, base_value
   ▼
[Stage 6] _assemble                   ← ParsedGear with field_confidences dict
   │
   ▼
Frontend review (color fields green/yellow/red) → POST /api/gear/manual
```

Stage 1 raises `TooltipNotFound` when neither strategy finds a plausible card;
the FastAPI ingest handler maps that to HTTP 422 with a useful error message.

Set `BLOOD_HUNT_OCR_DEBUG=1` to dump annotated PNGs of every stage's
intermediate output under `data/debug/<stage>/`. This is the primary tuning
tool — when the pipeline misclassifies, eyeball the debug images and adjust
the constants in the relevant module:
- `app/ocr/detect.py::MIN_AREA_FRACTION`, `ASPECT_*`, `CANNY_*`, `HSV_V_MAX`
- `app/ocr/anchors.py::_PROPORTIONS`, `_INK_FRAC_THRESHOLD`, `_MIN_ROW_HEIGHT_FRAC`

### 2.2 Stage 1 — Tooltip Card Detection

Two strategies run in order, both implemented in `app/ocr/detect.py`:

1. **Canny edge + external contour.** Convert to grayscale, blur lightly,
   Canny edges, dilate to close gaps, find external contours, filter by area
   (≥`MIN_AREA_FRACTION` of screen) and aspect ratio (`ASPECT_MIN..ASPECT_MAX`,
   default 0.6–3.0 — covers portrait-and-square-ish cards). Score by contour
   solidity (filled area / bbox area); higher solidity = more rectangle-like
   = more likely a tooltip. Returns the highest-solidity candidate above the
   threshold.

2. **HSV dark-background fallback.** When the game world has busy edges that
   swamp the tooltip border, threshold on low-V HSV pixels — the tooltip's
   semi-opaque overlay creates a large connected dark region. Same area +
   aspect filters; pick the largest valid candidate.

Both strategies emit annotated PNGs (`detect/canny_choice.png`,
`detect/hsv_choice.png`) when debug is on. Tune the constants by
inspecting these dumps.

### 2.3 Stage 2 — Proportional Anchors

`app/ocr/anchors.py::compute_anchors` returns a `CardAnchors` whose every
region is computed as a proportion of the **detected card's** dimensions —
never absolute pixels. Defaults in `_PROPORTIONS`:

```python
{
    "slot_icon":        (0.04, 0.04, 0.16, 0.14),  # top-left
    "name":             (0.20, 0.04, 0.60, 0.10),  # top centre
    "rarity_badge":     (0.82, 0.04, 0.14, 0.10),  # top-right
    "level":            (0.20, 0.14, 0.30, 0.06),  # under the name
    "base_effect":      (0.05, 0.24, 0.90, 0.10),  # divider band
    "extended_effects": (0.05, 0.36, 0.90, 0.55),  # bottom block
}
```

Each tuple is `(x_start, y_start, w, h)` as fractions of card width/height.
Tune by inspecting `anchors/regions.png` against real fixtures.

### 2.4 Stage 3 — Row Segmentation

`anchors.segment_rows` splits the extended-effects region into 0–5 row bands
by horizontal-projection minima. The number of rows is **detected**, not
assumed — supports tooltips with 0, 1, 2, 3, 4, or 5 extended effects without
prior knowledge of rarity (per-rarity caps: normal 0, advanced 1, rare 2,
epic 3, legendary up to 5).

Constants:
- `_GRAY_INK_CUTOFF = 80` — pixel intensity ≥ this counts as "ink" (tooltip
  text is light-on-dark, so high values are letters).
- `_INK_FRAC_THRESHOLD = 0.04` — a pixel-row is "text" if ≥4% of its pixels
  are ink.
- `_MIN_ROW_HEIGHT_FRAC = 0.06` — rows shorter than 6% of the region drop
  out as noise (single-pixel artefacts, separator lines, etc.).
- Hard cap at 5 rows.

Inspect `anchors/rows.png` to verify the right number of rows and where
they land.

### 2.5 Rarity Detection (Color-Based)

Tooltips border / name color encodes rarity. Sample the rarity badge centroid and classify by HSV hue. Vocabulary matches the in-game subtitles confirmed against user screenshots on 2026-04-27 (`NORMAL ARMOR`, `ADVANCED WEAPON`, etc. — earlier drafts used `common`/`uncommon`):

| Rarity | Hue range (approx) | Notes |
|---|---|---|
| Normal (white) | low saturation, high value | Detect via `S < 30 && V > 200` |
| Advanced (green) | H ≈ 60° | |
| Rare (blue) | H ≈ 220° | |
| Epic (purple) | H ≈ 280° | |
| Legendary (gold) | H ≈ 45°, high saturation | |

Tune on real samples — the values above are ballparks.

### 2.6 Preprocessing (per cropped region)

```python
import cv2
import numpy as np

def preprocess_for_tess(bgr: np.ndarray) -> np.ndarray:
    gray = cv2.cvtColor(bgr, cv2.COLOR_BGR2GRAY)
    # 2x upscale: Tesseract loves big text
    gray = cv2.resize(gray, None, fx=2, fy=2, interpolation=cv2.INTER_CUBIC)
    # Light blur, then adaptive threshold for non-uniform tooltip backgrounds
    gray = cv2.bilateralFilter(gray, 5, 50, 50)
    th = cv2.adaptiveThreshold(
        gray, 255, cv2.ADAPTIVE_THRESH_MEAN_C, cv2.THRESH_BINARY_INV,
        blockSize=31, C=10
    )
    # Invert so text is dark on light, which Tesseract prefers
    th = cv2.bitwise_not(th)
    return th
```

### 2.7 Tesseract Calls

Two configs, used per region type:

```python
import pytesseract

# For numeric values like "+8300%" or "Lv 60"
NUM_CONFIG  = "--psm 7 -c tessedit_char_whitelist=0123456789+-.,%Lv "

# For stat names ("Precision Damage", "Total Output Boost")
TEXT_CONFIG = "--psm 7"

# For single-letter tier badges (S/A/B/C/D)
TIER_CONFIG = "--psm 10 -c tessedit_char_whitelist=SABCD"

def ocr(img, config): return pytesseract.image_to_string(img, config=config).strip()
```

`--psm 7` = treat the image as a single text line. Right choice for tooltip rows. `--psm 10` = single character.

### 2.8 Fuzzy Normalization (content-based stat identity)

Tesseract will produce "Precision Oamage", "Tota1 Output Boost", etc. Normalize against the stat catalog from `gear_stats.json` (datamined in Part 1):

```python
from rapidfuzz import process, fuzz

def normalize_stat(raw: str, catalog: list[str]) -> tuple[str, float]:
    """Return (best_match, score). Score 0-100; reject below 75."""
    best = process.extractOne(raw, catalog, scorer=fuzz.WRatio)
    return (best[0], best[1]) if best else (raw, 0.0)
```

Values get a regex pass:

```python
import re
PCT  = re.compile(r"([+-]?\d{1,5}(?:[.,]\d+)?)\s*%")
LVL  = re.compile(r"L\s*v?\s*\.?\s*(\d{1,2})")

def parse_value(raw: str) -> float | None:
    m = PCT.search(raw)
    if m: return float(m.group(1).replace(",", "."))
    return None
```

### 2.9 Tier Letter Detection (Tesseract + template match)

Extended effect tiers (S/A/B/C/D, see RESEARCH.md §3.2) are tiny. Two strategies, run both and reconcile:

1. **OCR with `TIER_CONFIG`** on the cropped tier badge. The in-game UI wraps
   the letter in **full-width brackets `【S】` (U+3010 / U+3011)**; some
   screenshots may render as ASCII `[S]`. `app/ocr/parse.py::parse_tier_letter`
   strips both bracket families plus adjacent punctuation before scanning for
   the letter, so either form parses identically.
2. **Template match** against pre-saved S/A/B/C/D tier-icon crops (`cv2.matchTemplate`, `TM_CCOEFF_NORMED`). More robust at small sizes.

If both agree, confidence = 1.0. If only one, confidence = 0.65. If neither, surface to the user for manual correction. (0.65 keeps a single-method read inside the yellow band defined in CLAUDE.md §3.6 rather than landing on the yellow/red boundary.)

### 2.10 Output Schema

`apps/api/app/schemas/gear.py`:

```python
class BaseEffect(BaseModel):
    name:       StatId           # e.g. "Health", "Armor Value"
    value:      float            # numeric reading

class ExtendedEffect(BaseModel):
    stat_id:    StatId           # canonical name, post-fuzzy
    tier:       TierLetter       # S/A/B/C/D
    value:      float            # numeric reading (e.g. 8300.0 for "+8300%")
    raw_text:   str
    confidence: float            # 0..1

class ParsedGear(BaseModel):
    name:               str | None              # OCR'd item display name
    slot:               GearSlot
    hero:               str | None              # in-game display name, e.g. "Moon Knight"
    hero_id:            str | None              # canonical slug, e.g. "moon_knight"
    rarity:             Rarity                  # normal|advanced|rare|epic|legendary
    level:              int = Field(ge=1, le=60)
    rating:             int = Field(ge=0, default=0)  # tooltip overall rating
    base_effects:       list[BaseEffect]        # 1+ rows; armor has Health AND Armor Value
    extended_effects:   list[ExtendedEffect]    # max_length=5 (legendary cap)
    overall_confidence: float
    field_confidences:  dict[str, float]        # parallel dict — per-field 0..1
    source_screenshot:  str
```

`field_confidences` carries the per-top-field confidences (`name`, `slot`,
`rarity`, `level`, `rating`, `hero`, `base_effects`, `detection`). Per-row
confidences live on each `ExtendedEffect.confidence`.

### 2.11 End-to-End Pipeline (`apps/api/app/ocr/pipeline.py`)

```python
def parse_gear_screenshot(image_path: str, stat_catalog: list[str], *,
                          hero_id: str | None = None) -> ParsedGear:
    img = _read_image(image_path)
    card = detect_tooltip(img)                           # Stage 1
    card_bgr = crop_card(img, card)
    anchors = compute_anchors(card_bgr)                  # Stage 2
    extended_region = crop(card_bgr, anchors.extended_effects)
    row_bands = segment_rows(extended_region)            # Stage 3
    top = _extract_top(card_bgr, anchors, stat_catalog,  # Stage 5
                       detection_confidence=card.confidence)
    rows = [_extract_row(extended_region, band, stat_catalog)
            for band in row_bands]                       # Stage 4
    return _assemble(image_path, hero_id, top, [r for r in rows if r])  # Stage 6
```

`TooltipNotFound` from Stage 1 propagates to the FastAPI handler, which maps
it to HTTP 422.

### 2.13 Error Handling & Confidence Surfacing

The frontend shows the parsed result for review before save. Per-field colors
per CLAUDE.md §3.6:

- **Green** — `confidence ≥ 0.85`
- **Yellow** — `0.6 ≤ confidence < 0.85`
- **Red** — `< 0.6` (must confirm or correct before save)

The pipeline exposes `fields_below_review_threshold(parsed) -> list[str]` so
the frontend can show a single "needs review" banner regardless of how many
fields landed below the cutoff.

Every save records the source screenshot path so the user can re-parse later
when the pipeline improves.

### 2.14 Testing the OCR

`apps/api/tests/`:

- `test_detect.py` — synthetic full-screen images verify the detector finds
  embedded card-shaped regions and rejects out-of-spec geometry.
- `test_anchors.py` — proportional anchors and row segmentation against
  synthetic strips.
- `test_pipeline.py` — `resolve_tier` and `resolve_slot` reconciliation logic.
- `test_ocr_fixtures.py` — **the accuracy gate.** Loads every directory under
  `tests/fixtures/ocr/fixture_*/`, runs the full pipeline against the
  `screenshot.png`, and asserts the parsed result matches `expected.json`
  (numeric values within ±2% or ±1 absolute, stat order ignored). The
  aggregate test fails if pass rate drops below `TARGET_PASS_RATE = 0.9`
  per CLAUDE.md §7.1.

When fixtures are absent (early Phase 2), the fixture-gate tests are
skipped cleanly so `make test` stays green; they auto-activate as soon as
fixtures land.

## Part 3 — Patch-Day Runbook

Every Marvel Rivals patch (and especially every Blood Hunt content drop) follows the same checklist:

1. **Refresh AES key + mappings** from [Nexus Mods #1717](https://www.nexusmods.com/marvelrivals/mods/1717).
2. **Re-run FModel** exports for the DataTables in §1.4 → `data/game/_raw/`.
3. **`make extract`** to regenerate the canonical JSONs.
4. **Eyeball `version.json`** and the §1.8 sanity checks.
5. **Run OCR fixture tests** (`py -m pytest apps/api/tests/test_ocr_fixtures.py`).
   If accuracy drops, the tooltip UI has shifted — re-tune `app/ocr/anchors.py`
   `_PROPORTIONS` and/or re-capture template PNGs in `data/game/_assets/` per
   PHASE2_OCR_INPUTS.md. Use `BLOOD_HUNT_OCR_DEBUG=1` to see exactly what each
   stage produced.
6. **Bump app version**, commit, deploy locally.

A 10-minute drill, not a half-day re-engineering effort, as long as the runbook stays honest.
